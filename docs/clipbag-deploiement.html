Railway — Étape 1
Créer la base de données
On commence par la PostgreSQL. Railway la crée en 30 secondes et te donne une URL de connexion directement.
🗄️
railway.app
Créer la base PostgreSQL
  • 1
    Va sur railway.app → New Project → Deploy PostgreSQL.
  • 2
    Railway crée une instance PostgreSQL. Clique dessus → onglet Variables. Tu verras une variable DATABASE_URL générée automatiquement.
  • 3
    Copie cette DATABASE_URL — elle ressemble à :
    postgresql://postgres:motdepasse@containers-us-xxx.railway.app:5432/railway
  • 4
    Garde-la précieusement, tu en auras besoin dans les variables d'environnement de ton NestJS.
Prisma — connexion via DATABASE_URL
// prisma/schema.prisma — déjà configuré datasource db { provider = "postgresql" } // La variable DATABASE_URL est lue automatiquement par Prisma. // L'URL interne Railway fonctionne car backend + BDD // sont dans le même projet Railway.
💡
Le script npm run build du backend exécute automatiquement prisma generate + prisma migrate deploy avant de compiler NestJS. Les migrations sont appliquées à chaque déploiement.
Railway — Étape 2
Déployer le backend NestJS
Railway détecte automatiquement NestJS depuis ton GitHub et build le projet. Il faut juste lui dire comment le lancer.
⚙️
Railway
Déployer depuis GitHub
  • 1
    Dans ton projet Railway → New Service → GitHub Repo. Connecte ton compte GitHub et sélectionne first-ecommerce-site.
  • 2
    Railway va demander le Root Directory. Mets backend (c'est le dossier NestJS dans ton repo).
  • 3
    Dans Settings → Build Command : npm run build (inclut prisma generate + migrate deploy + nest build).
  • 4
    Dans Settings → Start Command : node dist/main
  • 5
    Railway injecte automatiquement la variable PORT. Le main.ts l'utilise déjà via process.env.PORT || 3000.
Vérifier ton main.ts NestJS
// backend/src/main.ts — déjà configuré correctement async function bootstrap() { const app = await NestFactory.create(AppModule, { rawBody: true, bodyParser: false }); // Raw body pour webhook Stripe, JSON pour le reste app.use('/api/payments/webhook', express.raw({ type: 'application/json' })); // ... parsers JSON/urlencoded pour les autres routes app.enableCors({ origin: process.env.FRONTEND_URL || 'http://localhost:4000', }); app.setGlobalPrefix('api'); await app.listen(process.env.PORT || 3000); }
💡
Railway injecte automatiquement la variable PORT. Assure-toi que ton NestJS l'utilise sinon le health check échoue et le déploiement boucle.

Vercel — Étape 1
Déployer le frontend Nuxt 3
Vercel détecte Nuxt 3 automatiquement. La seule chose à faire : pointer sur le dossier frontend de ton monorepo.
▲
vercel.com
Import du projet
  • 1
    Va sur vercel.com → Add New Project → Sélectionne ton repo first-ecommerce-site.
  • 2
    Ton repo est un monorepo — dans Root Directory → mets frontend.
  • 3
    Framework Preset : Nuxt.js (détecté auto). Build Command : nuxt build. Output Directory : .output.
  • 4
    Ajoute la variable d'environnement NUXT_PUBLIC_API_BASE = https://api.clipbag.fr/api avant de cliquer Deploy.
  • 5
    Clique Deploy. Vercel va builder et te donner une URL xxx.vercel.app — ça marche comme d'habitude.
✓
Nuxt 3 gère le routing SSR automatiquement sur Vercel — pas besoin de vercel.json. Vercel détecte le preset Nuxt et configure tout seul le routing, les API routes et le rendu serveur.
Domaine custom — Étape clé
Associer clipbag.fr à Vercel
C'est la partie que tu n'as jamais faite. C'est en réalité très simple — 3 DNS records à ajouter chez ton registrar (OVH, Namecheap, Gandi...).
▲
Vercel Dashboard
Ajouter le domaine
  • 1
    Dans ton projet Vercel → Settings → Domains → Add Domain.
  • 2
    Tape clipbag.fr. Vercel va te donner des DNS records à configurer. Note-les.
  • 3
    Ajoute aussi www.clipbag.fr et configure une redirection www → apex (Vercel le propose automatiquement).
🌐
Chez ton registrar (OVH / Namecheap / Gandi)
Records DNS à ajouter
TypeNom / HostValeur
A@ (ou vide)76.76.21.21 (IP Vercel)
CNAMEwwwcname.vercel-dns.com
💡
Vercel te donne exactement ces valeurs dans son dashboard. Si ton registrar ne supporte pas les enregistrements A (rare), utilise le CNAME fourni par Vercel à la place.
Propagation DNS — patience requise
  • ⏱
    La propagation DNS prend entre 5 minutes et 48h selon les registrars. OVH est généralement rapide (15-30 min).
  • ✓
    Vérifie la propagation sur whatsmydns.net — tape clipbag.fr et vérifie que le record A pointe vers 76.76.21.21.
  • 🔒
    Vercel génère le certificat SSL automatiquement (HTTPS) une fois la propagation terminée. Rien à faire de ton côté.

Stripe
Passer Stripe en production
En dev tu utilises des clés test (sk_test_). Pour vendre pour de vrai, il faut activer le compte Stripe et basculer sur les clés live.
💳
dashboard.stripe.com
Activer le compte Stripe
  • 1
    Dans Stripe Dashboard → Activate your account. Remplis les informations business (SIRET, RIB pour les virements, adresse).
  • 2
    Une fois activé → Developers → API Keys. Récupère la Publishable key (pk_live_...) et la Secret key (sk_live_...).
  • 3
    Mets sk_live_... dans la variable STRIPE_SECRET_KEY de Railway. Mets pk_live_... dans NUXT_PUBLIC_STRIPE_KEY de Vercel (si utilisé côté front).
🔔
Stripe → Webhooks
Configurer le webhook vers ton NestJS
  • 1
    Stripe Dashboard → Developers → Webhooks → Add Endpoint.
  • 2
    URL : https://api.clipbag.fr/api/payments/webhook
  • 3
    Événements à écouter : checkout.session.completed.
  • 4
    Récupère le Signing Secret (whsec_...) et mets-le dans STRIPE_WEBHOOK_SECRET sur Railway.
// Déjà implémenté dans payments.service.ts // Le webhook vérifie la signature Stripe automatiquement // et met à jour la commande (status PAID, adresse, email) // Route: POST /api/payments/webhook
